# Theming

The app ships 1000+ color themes that apply to both the React UI (CSS variables) and the Qt chrome (QSS stylesheets). Both sides are driven from the same source — `themes.json` — and stay in sync via the bridge.

## Architecture

```
themes.json (source of truth)
    │
    ├──► React: CSS variables on :root
    │    (applied by shared/lib/themes.ts)
    │
    └──► Qt: QSS stylesheets
         (generated by tools/generate-qss-themes.ts)
         (applied by StyleManager via qApp->setStyleSheet())

Bridge keeps them in sync:
  React ──setQtTheme()──► Qt
  Qt ──qtThemeChanged──► React
```

## Key Files

| File | What it does |
|------|-------------|
| `web/apps/main/src/data/themes.json` | 1000+ themes with light + dark CSS variables |
| `desktop/styles/shared/widgets.qss.template` | QSS widget map — how themes apply to Qt widgets |
| `desktop/styles/compiled/` | Generated QSS files (one per theme × mode) |
| `desktop/styles/compiled/theme-names.json` | Slug → display name mapping for bridge sync |
| `tools/generate-qss-themes.ts` | Generator: themes.json → QSS files |
| `desktop/src/style_manager.hpp/.cpp` | C++ theme loader, live reload, SCSS compilation |

## StyleManager

Loads and applies QSS themes. Three sources, checked in order:

1. **`STYLES_DEV_PATH`** (compile-time define, dev only) — points at `desktop/styles/` in the repo. Set automatically by xmake when not in CI.
2. **AppData/Local/\<app\>/styles/** — power user override folder. Create this folder and drop QSS/SCSS files in it to customize themes at runtime.
3. **QRC embedded** — `:/styles/<theme>.qss`. Pre-compiled, always available, fastest.

Sources 1 and 2 activate `QFileSystemWatcher` for **live reload** — edit a file, save, see the app update instantly. Source 1 also compiles `.scss` files at runtime via **libsass**.

### API

```cpp
// Apply a theme by full name (e.g. "synthwave-84-dark")
styleManager->applyTheme("synthwave-84-dark");

// Apply by base name + mode
styleManager->applyTheme("synthwave-84", true);  // dark

// Apply from a React display name (e.g. "Mrowr Purr - Synthwave '84")
styleManager->applyThemeByDisplayName("Mrowr Purr - Synthwave '84", true);

// Toggle dark/light — smart switching with fallback
styleManager->toggleDarkMode();

// Query state
styleManager->currentTheme();       // "synthwave-84-dark"
styleManager->currentBaseName();    // "synthwave-84"
styleManager->currentDisplayName(); // "Mrowr Purr - Synthwave '84"
styleManager->isDarkMode();         // true
styleManager->availableThemes();    // all full names
styleManager->availableBaseThemes(); // deduplicated, no suffixes
```

### Dark/Light Mode

Themes are named `<slug>-dark` and `<slug>-light`. When you toggle mode:

1. Tries `<current-slug>-<opposite-mode>` (e.g. `synthwave-84-light`)
2. If not found, tries the last remembered theme for that mode
3. Falls back to `default-dark` or `default-light`

Also calls `styleHints()->setColorScheme()` to update platform chrome (title bar, native dialogs).

## Qt ↔ React Sync

The bridge keeps both sides in sync. All methods take request objects (def_type DTOs), not positional arguments:

| Method | Direction | What it does |
|--------|-----------|-------------|
| `setQtTheme({displayName, isDark})` | React → Qt | React tells Qt to switch theme |
| `getQtTheme()` | React → Qt | Returns `{displayName, isDark}` |
| `getQtThemeFilePath()` | React → Qt | Returns `{path, embedded}` for current theme file |
| `qtThemeChanged` (signal) | Qt → React | Emitted with `{displayName, isDark}` when Qt theme changes |

**React → Qt:** `SettingsTab` calls `setQtTheme({displayName, isDark})` when the user picks a theme or toggles dark/light. The `main.tsx` startup code also syncs on load (React owns persisted state via localStorage).

**Qt → React:** The toolbar theme dropdown and dark/light toggle emit `themeChanged` on the `StyleManager`, which updates the bridge state and emits `qtThemeChanged` with a typed payload. A global listener in `App.tsx` catches this, applies the matching React theme, and dispatches a `qt-theme-synced` custom event so `SettingsTab` can refresh its UI.

**Name mapping:** Qt uses slugified names (`mrowr-purr-synthwave-84`), React uses display names (`Mrowr Purr - Synthwave '84`). The generator outputs `theme-names.json` with the mapping. `StyleManager` loads it on startup and auto-resolves in both directions.

**Sync guard:** A `qtSyncGuard` flag in `App.tsx` prevents infinite loops (React sets → Qt emits → React sets → ...).

## Generating Themes

```bash
bun run tools/generate-qss-themes.ts
```

Reads `themes.json`, applies the `widgets.qss.template`, outputs:
- `desktop/styles/compiled/<slug>-dark.qss` — one per theme
- `desktop/styles/compiled/<slug>-light.qss`
- `desktop/styles/compiled/theme-names.json` — slug↔name mapping

The compiled files are committed to the repo — no generator step needed after cloning. Re-run when:
- You edit `widgets.qss.template` (changes how colors map to widgets)
- You add/remove themes from `themes.json`

## The Widget Template

`desktop/styles/shared/widgets.qss.template` maps shadcn color tokens to Qt widget selectors. It uses `$variable` placeholders that the generator replaces with actual colors:

```css
QMenuBar {
  background-color: $background;
  color: $foreground;
  border-bottom: 1px solid $border;
}

QMenuBar::item:selected {
  background-color: $accent;
  color: $accent-foreground;
}
```

Variables available: `$background`, `$foreground`, `$card`, `$card-foreground`, `$popover`, `$popover-foreground`, `$primary`, `$primary-foreground`, `$secondary`, `$secondary-foreground`, `$muted`, `$muted-foreground`, `$accent`, `$accent-foreground`, `$destructive`, `$destructive-foreground`, `$border`, `$input`, `$ring`.

### Custom Widget Classes

The template uses Qt property selectors for semantic styling:

```css
QPushButton[class~="destructive"] {
  background-color: $destructive;
  color: $destructive-foreground;
}
```

Set these in your widget constructors:
```cpp
auto* btn = new QPushButton("Delete");
btn->setProperty("class", QStringList{"destructive"});
```

Available classes: `destructive`, `secondary`, `ghost`. Add your own in the template.

## Live Theme Editor

The Editor tab can load and edit the current QSS theme file:

1. Click **🎨 Edit Current Theme** — loads the active QSS file into Monaco
2. Edit colors, borders, whatever
3. **Ctrl+S**, **:w** (vim), or toolbar **Save** — saves the file
4. `QFileSystemWatcher` detects the change → theme updates instantly

The save action is context-aware: Ctrl+S and the toolbar Save button trigger the theme save only when the Editor tab is active and editing a theme. Otherwise, normal Save behavior.

## Adding a Custom Theme

1. Add your theme to `themes.json` with `light` and `dark` color objects
2. Run `bun run tools/generate-qss-themes.ts`
3. Both React and Qt pick it up — React from the imported JSON, Qt from the compiled QSS

Or for a Qt-only theme: drop a `.qss` file in `desktop/styles/compiled/` named `<slug>-dark.qss`. The StyleManager finds it automatically.

## libsass

The app embeds [libsass](https://github.com/mrowrpurr/libsass) for runtime SCSS compilation. This powers:
- Live reload of `.scss` files in dev mode
- Power users editing SCSS in the AppData styles folder

The build-time generator uses `npx sass` (Dart Sass) instead — it's faster for batch compilation and doesn't have the QSS pseudo-selector limitations that tripped up Dart Sass (the generator does direct variable substitution, not SCSS compilation).

## CI/CD

Three GitHub Actions workflows:

| Workflow | Trigger | What it does |
|----------|---------|-------------|
| `ci.yml` | Push/PR to main | Build + test (Catch2 + Bun + Playwright) on Windows + macOS |
| `release.yml` | Tag push (`v*`) | Build + test + `windeployqt`/`macdeployqt` + GitHub Release |
| `nightly.yml` | Daily 6AM UTC | Same as release, but only if commits in last 24h. Rolling pre-release. |

All three cache Qt, xmake packages, and Playwright browsers. Playwright runs directly (`npx playwright test`) instead of through xmake because `os.execv` can't resolve `.cmd` shims on Windows.
